银科环企GIT规范

背景

  • 使用Git管理一样需要标准化实施细则，开发人员、测试人员以及运维人员如何使用：

    1. 创建仓库
    2. 标准化项目分支结构
    3. 控制源码管理
    4. 标准化开发，发布流程
    5. 保证任何情况出现，都能快速恢复当前生产环境，迅速回滚到上一个稳定的版本

 

创建仓库

  1. 群组结构：

    • 创建群组以及仓库由Git管理员执行

分支规划

  • master主分支(全生命周期,只允许合并)
  • uat_**-release分支(uat测试分支,全生命周期,允许提交以及合并)
  • **_**-release分支(线上分支,全生命周期,允许提交以及合并)
  • last_**_**-release分支(上一个稳定的线上版本分支,全生命周期,只允许合并)
  • dev_**分支(短生命周期,允许提交以及合并)

操作流程

案例相对于需要进行uat测试的项目**(以中南项目ota3为例)

  1. 前期准备

    1. 创建项目,运维人员在master分支编写初始.gitlab-ci.yml文件
    2. 从master拉取zhongnan_ota3-release分支
    3. 从zhongnan_ota3-release拉取last_zhongnan_ota3-release分支
    4. 从zhongnan_ota3-release拉取uat_ota3-release分支

  2. 开发

    1. 从uat_ota3-release上拉取最新代码,命名为dev_ota3.1(需求或功能名),已经拉取则忽略这一步
    2. 完成开发并自测之后,填写commit信息,然后提交代码(只需提交,不需push)
    3. 切换到uat_ota3-release,合并dev_ota3.1分支,打上版本信息tag:uat_ota3.1_1.0(非必须),然后push

  3. uat测试

    1. 开发人员合并uat_ota3-release分支之后,会自动推送企业微信的消息,测试人员点击连接,可以部署在uat机器上,测试人员随后进行测试
    2. 部署出错,运维人员检查cicd文件,重新推送微信消息,测试人员重新执行操作(稳定之后几乎不会出现)
    3. 测试不通过,测试人员通知开发人员修改,回到第2步继续走流程
    4. 测试通过,测试人员告知开发人员

  4. 线上发布

    1. 开发人员收到测试通过的通知,切换到zhongnan_ota3-release分支,合并dev_ota3.1分支,打上版本信息tag:zn_ota3.1_1.0(非必须),然后push
    2. cicd会自动将当前版本的zhongnan_ota3-release分支备份到last_zhongnan_ota3-release分支,并自动发布
    3. 发布出错,通知运维人员检查将last_zhongnan_ota3-release发布,检查cicd文件,再重新发布(稳定之后几乎不会出现)
    4. 线上出现小型bug,执行2,4操作
    5. 线上出现大型bug,通知运维,将last_zhongnan_ota3-release发布,开发人员则执行2,3,4操作,修复bug

    连续两次出现重大bug要注意,此时last_zhongnan_ota3-release已经不是稳定分支,此时请通知运维进行手动回滚

    对于个别不需要uat的项目,将第三步剔除即可
